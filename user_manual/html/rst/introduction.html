Introduction¶

This manual describes the usage instructions for CDH/ACE (further referred to as ACE), a software framework to perform Autonomous Computational Experiments using Nastran and/or Optran based FEM solvers. At the time of writing, ACE is mainly focussed towards, but not limited to, applications in the field of Noise and Vibration Harshness (NVH).

A pdf version of this document is available under section Download pdf versions.

Basic idea and concept¶

The idea of ACE is to perform Autonomous Computational Experiments for CAE simulation. Similar to how engineers typically proceed, the basic principle consists of the following 4 elementary steps:

  • try out different CAE simulation/model experiment parameters (e.g. design variables)

  • observe/remember the influence on the CAE results / target settings

  • learn correlations or identify patterns between experiment parameters and results

  • predict/take action to modifythe parameter values such that a target is reached

By automating the above steps a self-learning loop is created that tries to reach a specified target setting in an autonomous manner by experimentation (see Fig. 1).

Within the context of CAE analysis this idea can be translated to a flow schedule as illustrated in Fig. 2.

On the left-hand side of the flow schedule, the conventional CAE process is shown where a physical model is made and then solved to predict the responses of the system under investigation. This side is often referred to as direct modelling because the underlying physical laws are directly captured by the numerical model. Within the field of CAE, the finite element method (FEM) has evolved to one of the most important approaches for function validation during product development.

Whereas on the right-hand side, in opposing direction, a so-called data model is made that relates the computed responses of the physical model with the original input parameters. This is usually referred to as indirect modelling as the resulting model is only based on observations of input and outputs of an otherwise unknown system (without describing any physics). Nowadays, most of the data models are based on machine learning (ML) and artificial intelligence (AI) approaches.

One major advantage of the direct physical modelling is that is reveals insights in the physical mechanisms and allows for identification of root causes and effects for potential problems or questions at stake. The down-side however is that it typically requires quite some experience to setup a meaningful CAE simulation and that the actual solving of the model can take up quite some time and resources (e.g. several hours for a single run).

In contrast, the computation time and associated resources to make a prediction using an available data model (e.g. using a trained machine learning model) is extremely fast (often on the order of miliseconds). As a consequence such an model can be used easily for predicting system-behavior for many variations of the input parameters or for using such a model in a real-time environment. A futher advantage of such models is that they can be used to predict behavior of systems that have too complex behavior or have no obvious underlying physical laws, e.g. weather predictions heavily rely on observations. Drawback is that these techniques require many more-or-less reliable observations of the input and output data.

Finally, the idea behind ACE is to couple the world of engineering science (physical models) with the ML/AI technologies from the field of data science in a highly automated and user friendly way. This results in a self-learning and auto-correcting autonomous system that is able to setup it’s own CAE simulation runs by modification of specified input parameters, submit the CAE jobs into a HPC cluster, collect the response data, train machine learning models and use these in an AI-agent to setup further CAE simulations to reach a user specified goal.

Areas of applications and key features¶

Typical areas of application of ACE consist amongst others of:

  • handling of advanced CAE processes (e.g. loadstorm, mode agent plugins)

  • automated design of experiments (DOE)

  • Parameter optimization and/or identification

  • Sensitivity analysis and robustness studies based on fast evaluation of ML-models

  • Model reduction using trained ML models (real-time models)

  • Python based integration and automation of customized tools / plugins

  • Performance optimization of simulation and solver settings

Some of ACE its key features are:

  • Combine machine learning algorithms with efficient CAE solvers

  • Ability to generate it’s own CAE result data where required (autonomous system)

  • Self-learning and self-correcting nature due to iterative learning cycles

  • Focused toward efficient use for discipline of vibration and acoustics

  • Fully automated interface with Nastran, Optran and/or Optistruct FEM solvers

  • Easy to use, minimum man-power required to setup

  • Highly scalable across many CAE hosts / compute nodes

  • Supports up to 64 CPU-cores per host

  • Able to dock on to existing CAE queueing systems

  • All relevant data and post-processing pictures are generated fully automated

  • Modern and open python based concept allows easy integration of own customized tools and plugins

  • Ideal as optimization and robustness study solution for Optran

Prerequisites and Limitations¶

Following prerequisites and/or limitations apply to the ACE framework

  • Officially supported hardware consists of Intel-Xeon or AMD-EPYC based HPC systems

  • The host that runs ACE itself should have at least 64Gb of memory and have a minimum of 4 physical CPU cores (recommended: 8 cores or more)

  • The IT-ecosystem should be based on 64bit Linux systems (kernel version > 3.10)

  • At least one of the following FEM solvers is required to perform the CAE simulations:

    • Hexagon / MSC Nastran

    • Simcenter / NX Nastran

    • Altair Optistruct

    • CDH/Optran

  • If the CAE solver environment is controlled through a task queuing / submission system ACE can take advantage of that to distribute the CAE jobs across multiple compute nodes/hosts (full DMP distributed computing)

  • Unpacking of the ACE installation archives must be performed on Linux as symbolic links are available that are typically not supported on Windows OS.

  • Some of job-settings are currently prescribed by ACE and are not yet allowed to be changed through a configuration file. This functionality to generalize settings will be added successively with the upcoming ACE versions
