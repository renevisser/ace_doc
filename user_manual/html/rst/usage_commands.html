Commands and modes of operation¶

Main commands and plugins¶

The ACE framework contains, besides the main program itself, of the following modular plug-in tools that may interact with ace and/or each other. Note that depending on the installation the actual command names can differ from those listed here. Please refer to your IT adminstrator if these commands are not available on your terminal.

  • ace: CDH/ACE command to perform autonomous computational experiments

  • py_ace: embedded python environment (version 3.10 incl. relevant packages)

  • Several ACE plugins:

    • loadstorm: loadstorm procedure for mode agent and mode agent2 plugins

    • mode_agent_single: mode agent to process results for single experiment variant

    • mode_agent_collect: mode agent to collect mode agent data from all experiments

    • mode_agent2_single: mode agent2 to process results for single experiment variant

    • mode_agent2_collect: mode agent2 to collect mode agent data from all experiments

    • vao_agent: perform easy setup of CDH/VAO analysis based on an ACE project

    • robustness_study: perform robustness studies and fast response predictions using trained ML-models

This manual concentrates on the main ACE application, whereas the general plugins are described in a separate section plugins. Note the embedded python version for ACE can be accessed by typing:

# command to start ace python version:
py_ace

This python distribution contains many relevant packages for advanced data science applications (see ace license --opensource for versions of included packages). It is highly adviced to use this provided python version for writing custom plugins or add-ons. Please contact CDH-support in case of further information on writing custom plugins is required.

Modes of operation (sub-commands)¶

This section concerns the various operation modes and their options available for the main ACE command. As common with more complicated command line tools, ACE also offers several modes of operation (also referred to as sub-commands) that are used to bundle related or distinct tasks. In general each operation mode of ACE offers a brief command line help function to get some quick information how to operate the specified mode. Independent of the mode of operation this help can be accessed by specification of option -h or --help.

Main command¶

Apart from the standard help -h functionality, the only other option available to the main ACE command itself is the -v or --version option to query the version number.

Requesting help of the main ace command:

# get a brief help function for the main ace command:
ace -h  # (alternative: --help)

will display the available operation modes (positional arguments) as well as an overview of the options available the the command:

Sub-commands (modes)¶

In total 4 distinct task-specific sub-commands, or “modes of operation”, are available:

  • new: setup a new ACE project

  • run: start a run of an existing ACE project

  • doc: access documentation and examples

  • license: options for license related operations (check, update, …)

A sub-command is always positioned directly after the ace command but before the mode specified options according to the following semantic scheme:

# semantic scheme:
command subcommand option[s]

# example to invoke "license" mode and perform option "check":
ace license --check

Note that each sub-command has its own mode-specific usage help function by specification of option -h. In addition it is emphasized that options always support the short format (single dash “-”) and an equivalent long format (double dash “–“), e.g. ace license -h is equivalent to ace license --help. Whether short or long format should be used depends on the use-case (e.g. usage in a script or direct user input on the command line) and personal user preference.

For the sake of completeness the available options for each of the sub-commands / modes are described in the remainder of this session but in practice the build-in usage help -h will be more convenient and quicker to access.

Sub-command new¶

The new mode bundles all options to setup a new ACE project and is accessed using the sub-command new. This is typically the starting point of any new ace study. Before starting any ACE project, it is assumed that the user has a properly working Nastran or Optran job available that serves as reference job for all further downstream ace investigations. This new mode will read and parse the reference job and will use it to setup the so-called initial baseline job-directory. The mode-specific usage help function gives an overview of the available options:

# show usage help for setup of new ace project:
ace new -h

yields (header lines are blended out for brevity):

usage: ace new [-h] [-n NAME] [-r REFERENCE] [-o OUTPUT] [-e ENCODING]
               [--encoding_errors {strict,ignore}] [--verbose]

options:
  -h, --help            show this help message and exit
  -n NAME, --name NAME  name of new ace project
  -r REFERENCE, --reference REFERENCE
                        reference solver input deck file (Nastran)
  -o OUTPUT, --output OUTPUT
                        output path to create the new project
  -e ENCODING, --encoding ENCODING
                        Text encoding scheme to apply in project (default:
                        utf8)
  --encoding_errors {strict,ignore}
                        Encoding errors handling (default: strict)
  --verbose             verbose mode: echo debug information to the
                        console/log-file

To create a new ACE project based with the name my_example based on a valid Nastran or Optran reference input deck input_deck.nas located in path path_ref:

# create new ace project based on reference input deck:
ace new --name my_example --reference path_ref/input_deck.nas

# or alternative specification in short format:
ace new -n my_example -r path_ref/input_deck.nas

In case of Optran input be sure to specify the input deck and not the so-called Optran “request” file (a potentially available request file will be recognized by ace automatically).

Option --output can be used to specify a different output path to write the new project to (other than the default current work directory).

Option --encoding should be used for cases where the input deck contains non-ASCII standard characters to specify the according text encoding scheme. For example to use models with Japanese characters use --encoding_errors shift-jis or, likewise, for models containing German special characters --encoding_errors latin1 should be applied. Errors concerning incompatible encoding typically can be solved by application of the correct scheme. Note that mixing different encondings within a single input deck is highly disadviced will likely cause further downstream difficulties. However, to use such a mixed setup ACE can be configured to ignore all encoding errors by sepcification of --encoding_errors ignore. The default encoding scheme is typically set during the installation of ACE in the ace_default_settings.toml configuration file. For configuration of ACE defaults reference is made to the installation manual.

After invoking ace new ACE will start reading, parsing and copying the provided reference model into the so-called ace project directory “my_example_ace” which consists of the project name my_example with the suffix _ace. Note the suffix _ace should not be removed from the project directory as ace needs the suffix for identification purposes (in general do not manually rename projects afterwards). Furtermore a standard directory structure and amongst others a so-called ace project file is generated in the project directory. The purpose of the “*.ace” project file is used to configure the ace simulation setup, e.g. similar to the funciton of a Nastran input deck. From this point onwards, ACE no longer needs the original specified input deck as it made a (restructured) copy of the reference simulation job into the baseline sub-directory. All further ACE steps use this baseline job/model as reference. Note the project directory as a whole can be copied to any other location without compromising the vadility of the ace job (all references are enclosed internally within the ace project directory). For a more detailed description of the ACE directory structure reference is made to section Project directory structure.

Sub-command run¶

The run mode bundles all options to start an existing ACE project and is accessed using the sub-command run. The main purpose of this mode is to either start a “fresh” ace run (e.g. just created with ace new) or to continue an existing ace run by invoking the restart option. It is recommended to execute the ace run command from within the ace project directory (typically ends with suffix “_ace”). All ace results are written the project directory.

The built-in usage help function gives an overview of the available options:

ace run -h

yields (header lines are blended out for brevity):

usage: ace run [-h] [-r] [-s SAMPLING] [-c {random,center}]
               [-f FILE_DOE_SAMPLES] [-n N_SAMPLES_PER_DIMENSION]
               [-b N_BATCHES_MAX] [--verbose]
               path

positional arguments:
  path                  path to /in ace project directory

options:
  -h, --help            show this help message and exit
  -r, --restart         use restart mode to continue previous analysis
  -s SAMPLING, --sampling SAMPLING
                        DOE sampling type, specify either: univariante,
                        fullfactorial, lhs or random
  -c {random,center}, --criterion {random,center}
                        criterion for LHS sampling within each interval
                        (default: None)
  -f FILE_DOE_SAMPLES, --file_doe_samples FILE_DOE_SAMPLES
                        Supply initial DOE samples for batch00 using a csv
                        file
  -n N_SAMPLES_PER_DIMENSION, --n_samples_per_dimension N_SAMPLES_PER_DIMENSION
                        number of LHS samples per non-constant experiment
                        parameter/dimension/feature
  -b N_BATCHES_MAX, --n_batches_max N_BATCHES_MAX
                        maximum allowed number of batches (1 implies a
                        conventional DOE)
  --baseline            run baseline simulation only (alias for "-b1 -n0")
  --verbose             verbose mode: echo debug information to the
                        console/log-file

The input argument “path” is mandatory and refers to the ace project directory (typically the current work directory if ace executed inside the project directory).

To make a quick reference run of the baseline model only, e.g. at the beginning of an ACE analysis to validate whether the target has been defined correctly, the option --baseline is available. This is a simple alias for options -b1 -n0 (run single batch without any variations) and that ignores all DOE settings.

Often it is useful to continue an existing ace run using the restart option --restart, e.g. to find a beter optimum or to continue after some CAE job might have failed.

The option --file_doe_samples can be used to specify values for the initial design of experiment samples from an external file instead of using the standard ACE provided DOE functionality (see section design of experiments).

The remaining options are convenience options in case the user would like to quickly override some of the settings in the ace project file (*.ace) without modifying the file itself.

Finally, the option --verbose might be helpful for debugging purposes (e.g. providing extended log information to CDH support).

Note

Specified command line options override potential equivalent settings from the ace project file (command line options have highest priority).

A typical ace run command would look like

# start a new/fresh ace run from within the project directory:
ace run .

# or continue an existing ace run and override batch
# setting to use a maximum of 10 batches:
ace run --restart -b10 .

A full description of all options for run mode is given in section ace project file. Furthermore, the ace run mode and its options is learned best by running some of the tutorials from the example manual.

Sub-command doc¶

The sub-command doc handles all options for accessing documentation as well as extracting reference models for the tutorials described in the example manual. As for each mode, -h or --help provides an quick overview of the available options:

# brief overview of access to the documentation
ace doc --help

results in:

========================================================================
                             ACE 2025.006e
                  Autonomous Computational Experiments
========================================================================
                    Visit us: https://www.cdh-ag.com
                 Contact us: mailto:support@cdh-ag.com
                 Copyright CDH AG. All rights reserved.
========================================================================

usage: ace doc [-h] [-o] [-c] [-e EXTRACT_EXAMPLE]

options:
  -h, --help            show this help message and exit
  -o, --open            Open main documentation (html)
  -c, --copy            Get a local copy of the documentation (html/pdf)
  -e EXTRACT_EXAMPLE, --extract_example EXTRACT_EXAMPLE
                        Extract / get example Nastran model for tutorial
                        (example manual)

In case of linux based platforms, the recommended way to access the documentation is to open the html based version in the system default browser using option --open (or -o). Alternatively, a local copy of the documentation in both html and pdf format can be obtained using the --copy option, resulting in a sub-directory ace_manual in the current work directory.

Other useful option is to extract the Nastran/Optran CAE example jobs that are used throughout the tutorials in this manual. To do so, the name of the example reference job, e.g. ms2dof, should be specified along with option --extract_example in following manner:

# brief overview of access to the documentation
ace doc --extract_example ms2dof

Which creates a local copy of the specified reference job in the current work directory for use in the corresponding tutorial. In case a non-existing example name is provided a list of available reference job is displayed.

Sub-command license¶

The ACE license mode bundles all license specific operation and is accessed using the sub-command license. Typically this mode is of interest during the installation of ACE or in case license issues are experienced. The built-in usage help function gives an overview of the available options:

ace license -h

yields (header lines are blended out for brevity):

usage: ace license [-h] [-c] [-u UPDATE] [-o]

options:
  -h, --help            show this help message and exit
  -c, --check           check license status
  -u UPDATE, --update UPDATE
                        Update license file
  -o, --opensource      Show license info of open source packages
