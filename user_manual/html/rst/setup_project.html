Setup a new ACE project¶

This section describes some important definitions required to understand the remainder of this manual as well as the example manual. In addition, an explanation how to setup a new ACE project and what the resulting directory structure looks like.

Definitions and terminology¶

In order to use and understand ACE effectively, this section explains most relevant domain-specific terminology and definitions that are applied throughout the program and its manuals.

Each ACE analysis is associated with an ACE project that is basically the overal accumulation of all project relevant data and settings, including the FE-models that are used in the analysis. All data is gathered in the the so-called ACE project directory. This directory encloses all data such that an ace project directory can always be archived (e.g. zipped) and copied or send to any other location without compromising the ability to rerun or restart the ACE analysis. The ace project file contains all settings for the ace run and hence it can be seen as the central “input deck” for ace. The ace project directory always ends with suffix ${project_name}_ace whereas the ace project file is `` indicated with file extension ${project_name}.ace, where ${project_name} is the user specifed name for the ace project.

A batch is a collection of Nastran/Optran simulation variants/jobs that is computed within a single learning cycle of ACE (see overal flowschedule in Fig. 3). All data for each batch is gathered in a batch directory, identified by name batchXX where two digits XX represent the number of the batch. The first batch, batch00, typically contains the initial Design of Experiments (DOE) that forms the first set of experiments for the subsequent machine learning algorithms.

An individual CAE/Nastran job is called a simulation variant (sv), it is stored in a job-directory, and typically contains a single experiment sample, often denoted as just sample. Each sample represents an unique experiment, a unique set of experiment parameter values, from the batch of experiments. In engineering sciences, experiment parameters are often named design variables, e.g. thicknesses of parts. In ACE however, experiment paramters can be more much general, for example if the topic of interest would be solver performance they could also represent quantities like the number of CPU-cores or memory settings applied in the simulation. Also external environment variables can be used as experiment parameters basically allowing to enable to use every command line tool inside an ACE analysis. For this reason ACE adapts to the more general naming of experiment parameters or -more common in the field of data science- features. Both the terms experiment parameter and feature are used interchangeably in ACE. From time to time also the more specific design variable wording can be used in case the parameters describe design related features.

Note

In case certain applied terminology is unknown or unclear, it might be of help to consult the list of abbreviations.

Create a new project¶

Before actually starting any simulations with ACE, the first step consists of defining a new ACE project. As explained in section sub-commands, this is easily accomplished using the sub-command ace new which automatically creates a new ACE project based on a user specified existing Nastran or Optran simulation input deck which serves as reference for new ACE project. It is mandatory to define a name for the project using the --name input argument:

# create new ace project based on reference input deck:
ace new --name my_example --reference path_ref/input_deck.nas

# or alternative specification in short format:
ace new -n my_example -r path_ref/input_deck.nas

This will result in a new ace project directory called my_example_ace (mandatory suffix _ace is added automatically). The project directory is intended to be completely “self-supporting” without any dependencies outside of it. This is, amongst others, useful for archiving and/or moving to any other path or location without compromising the ability to restart the ACE analysis run.

For more detailed input options for ace new please refer to section sub-commands new.

Project directory structure¶

During project setup ACE reads the so-called reference job and creates an ACE project directory where all dependencies of the reference job, including the FEM model itself, are collected. In addition, a couple of predefined sub-directories are created resulting in the following main project directory structure:

$ # show 2 level directory structure of ace project directory (after "ace new"):
$ tree -d -L 2 my_example_ace

my_example_ace
├── baseline
│   ├── bdf_csv_info
│   └── incl1
├── batch00
│   ├── experiments
│   └── templates
└── results

The baseline directory contains a -restructured- copy of the specified Nastran reference model and will be used as reference or baseline variant for all further ACE analysis within the project. Note that the baseline model is the unmodified original design which contents will never be changed by ace. In most cases inside the baseline directory, a sub-directory incl1 will be available that contains all the level-1 include files that were found in the main Nastran input deck. In case nested include files were present also higher level include directories like incl2, incl3, etc… will be available. Currently ACE allows up to a maximum of 9 levels deep nested include files. Furthermore a sub-directory bdf_csv_info is available that contains -compressed- comma separated files (csv) with parsed bulk-data model information. These csv files are used by ACE to quickly query model information that is required in later steps of the ace analysis. Also for some users these csv maybe helpful to use in their own custom processes, e.g. filtering of solver card entries in tools like Pandas or Microsoft Excel.

Directory batch00, also referred to as the batch directory, is intended for the first batch of experiments (simulation jobs) which are typically generated at a later stage by a design of experiments (DOE). The experiments sub-directory is intended to store the model changes for each experiment sample / Nastran simulation job inside the current batch directory. Note that ACE only stores the changes of the model for each experiment / simulation variant thereby significantly reducing the required amount of disk space compared to storing the full model for each experiment. Finally the sub-directory templates contains the template file that with generic solver cards used as experiment parameters (features) in the ace run. Obviously, these sub-directories are merely placeholders until the ACE run is configured and started (using ace run). During the the later ACE run, a new batch directory will be created for each learning-cycle. The batch directories are numbered monotonically increasing with a 2 digit scheme, e.g. batch01, batch02, …

Initially, the results directory is an empty placeholder for storing all the ACE results to be generated at a later stage in the process.

In addition to the project and its sub-directories, also the so-called ace project file is generated, here my_example.ace. This project file is used to control and configure the ace run, ranging from definition of the experiment parameters, setup of CAE submit commands, response and target definitions up to controlling the applied machine learning settings. For a detailed description of the ace file reference is made to section ace file.
