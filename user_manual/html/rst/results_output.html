Results¶

One of the key features of ACE is that a lot of post-processing is completed automatically during the ace run. The intension is to create the most relevant result plots and/or csv tables without the need of the end user to spend much time on such mainly repetative tasks. In this section the most important plots and csv output tables are explained with focus on, but not limited to, design optimization.

Plot descriptions (png files)¶

A brief desciption for each plot is listed in this section. Results are typically stored in the results sub-directory of the ace main project directory, altough some more detailed and batch (learn-cycle) specific plots are available in the batch sub-directories as well. For more details on the directory structure reference is made to section directory structure in chapter setup a new project.

Baseline response and target¶

At the beginning of an ACE run, it is typically of interest to plot the user-specified target definition versus the unmodified baseline response to verify whether the target has been setup correctly. The recommended and quickest way to perform this is to let ACE run only the baseline simulation using the command ace run -b=1 -n=0 (this sets up 1 batch with 0 experiments, meaning only the unperturbed baseline response is computed). At the end of this baseline run, a response plot for each target definition is made. Note that if multiple experiments are made (e.g. -n > 0) in the initial batch, then also the worst and best response curve of the DOE is added to the plot (measured using the target violation). See section sub-command run for further information on the command line options available while starting an ACE run.

Optimized response¶

The optimized response plot gives an overview of the initial baseline response, the best response in the initial DOE (batch00) and the response after optimization. Furthermore it typically shows the specified target curve and highlights the area where a target violation remains for the optimized response. Note that for convenience also a symbolic link link_to_optimum is created to the job-directory that contains the results of the optimized response.

Worst-case response¶

This plot is similar to the optimized response plot but in addition shows the worst-case response with the maximum target violation. Furthermore is displays the area in grey of all available response curves. This plot provides an indication about the variance of the responses and hence, the robustness of the system. Like for the optimized refesponse, also for the worstcase response a link link_to_worstcase is created for easy access to the corresponding job-directory.

Convergence history over batches¶

The convergence plot is important in optimization runs (number of batches / learn-cylces > 1) and shows how the solution converges to the specified target. All solution is known to be “converged” once the target violation becomes zero (target reached) or certain convergence criteria are met (e.g. number of batches without convergence has been reached). Depending on the problem at hand, sometimes it makes more sense to look at the convergence on a linear scaling (left-hand side) whereas for other tasks it might be more appropreate to look at the logarithmic convergence (right-hand side).

A more detailed convergence plot is also available that more-or-less shows the same information as the overal convergence plot but for each machine learning (ML) model individually. This detailed plot also gives an indication how well each of the ML algorithms was able to capture the system behavior.

Best response history over batches¶

Whereas the convergence plot provides information on the target violation, the best response plot shows how the response itself evolved for each successive batch in the optimization run. Again, the best response is measured with respect to the minimum target violation.

Feature history over batches¶

Similar to the best response over the batches, this plot shows the history of the best feature values (experiment parameters) over each batch. Only the top 12 most important features are shown, sorted by their descending importance (first feaature sub-plot is most important according to the ML-models).

Feature comparison baseline to optimum¶

This plot gives an overview of the change in feature values between the baseline and the optimized design state. The first sub-plot compares the feature values, the second and third sub-plot the absolute and relative difference and the last sub-plot contains the local importance of each feature. The local importance measures the contribution of each feature to go from the baseline to the localized optimum. Again, only the top 12 ranking of most important features are displayed.

Feature importance (local and global)¶

The meaning of feature importance is often a source of confusion as it depends on the context in which it is used. In ACE, a distinction is made between so-called local feature importance and global feature importance.

local importance:

a measure of the feature contribution in reaching the optimized response (a local point in the design space)

global importance:

a measure of the influence feature has on the target violation thorughout the whole design space (a global measure)

It should be pointed out that both importance criteria heavily dependent on the underlying ML-models that are used for predicting the importances. To derive an ‘overall importance’, ACE uses a weighted combination of both the local and global importances of each predictor. This overall averaged importance is then used to determine the top feature rankings to filter the most relevant features in several of the other plots.

Response surface predictions¶

This series of plots show the response surface as a slice though the optimum as predicted by each of the ML-models. The black circle represent the (projected) baseline location whereas the optimum is indicated by the cross marker. Based on the weighted feature importance only the slices along the top 3 feature are shown. Be careful while interpreting the response surfaces as they are only as accurate as the ML-model is able to predict them.

Ranking samples plot¶

Ranking samples are requested by specification of a value N larger than 0 for sampling approach ranking. This strategy then tries to find the optimum set of N features that results in the lowest target violation while keeping all other features constant at their baseline value. E.g. if a total of 10 features are available, a ranking value of 2 results in the change of only those 2 features that result in the lowest target violationl (best possible solution). The associated ranking samples plot shows which top N features have been selected and how changing their value from baseline to the optimum value successively contributes to lowering the target violation. Goal is to assist in identification of an optimal subset of features that still yields an adequate response. For stability reasons, the ranking plot is created only after 4 or more batches are available (so earliest in batch03 a ranking plot is available).

Prediction errors¶

In order to get an indication how well the different ML models are performing two further accuracy related plots are exported, the training and the prediction error plot.

training error:

accuracy history of the cross-validation error over the batches. Generally, the quality of a model its prediction increase with decreasing cross-validation error. Furthermore, an asymptotical behavior of the error shows taht dding more samples does not further increase the quality of a model. In this plot e.g. ‘linreg’ does no longer benefit whereas ‘gpr’ keeps improving while adding more batches (more samples).

prediction error:

accuracy history over batches indicating how well each of the trained ML models is able to predict the target violation at the optimum.

Feature pair plot¶

The pair plot shows all samples versus the features and target violation. Goal is to gain an understanding of correlations between features and the target violation in dependence of the sampling approach. At most the top 12 importance ranked features are displayed (blend out non-relevant features).

Table descriptions (csv files)¶

For easy access using spreatsheet tools or scripting languages, certain relevant data is exported to comma separated file format (*.csv). In case the csv filename matches the name of a plot/image file the contents will reflect the information available in the plot. However, also a number of additional csv files are generated that do not belong to a plot directly.

Mass tables (baseline and optimum)¶

During Nastran model parsing ACE computes the mass associated with all known solver cards for the initial baseline state and if available also for the optimized design state. Besides computing the overall mass, the mass is also aggregated (collected) on property type and property identifier level:

-mass_table_baseline_ptype.csv: mass table for baseline model, aggregated per property type

(e.g. mass of summed over all PSHELL cards)

-mass_table_baseline_pid.csv: mass table for baseline model, aggregated per property

identifier (e.g. mass summed per PSHELL card ID)

-mass_table_optimum_ptype.csv: mass table for optimized model, aggregated per property type

(e.g. mass of summed over all PSHELL cards)

-mass_table_optimum_pid.csv: mass table for optimized model, aggregated per property

identifier (e.g. mass summed per PSHELL card ID)

Note that the last row/line in the mass table always contains the total summed mass in the table. Furthermore, for some elements that are commonly used but have no property associated with them, a dummy property “PDUM” is created to enable displaying of these mass values as well (e.g. PDUM_CONM2 aggregates the mass of all CONM2 elements).

Likewise, the csv file mass_table_baseline_pid.csv contains the mass aggerated for each poperty identifier (pid) separately, by introducing an additional pid column. These tables are often useful to get a brief overview about the mass in both the baseline state as well as the optimized design state.

Experiment samples¶

At the end of an ACE run, a large table is exported in the csv file experiment_samples.csv under the results directory. This table contains amongst others the feature values for each of the samples that has been generated in every batch of the run. Furthermore information about the target violation (res column) and the batches and sampling appoach is contained.
